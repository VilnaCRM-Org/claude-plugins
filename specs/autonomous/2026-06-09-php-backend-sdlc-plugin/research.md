## 2. Claude Code plugin packaging constraints

- **marketplace.json**: `{name, description, owner{name,email|url}, plugins:[{name, description, author, category, source, homepage}]}`. `source` is either a relative path string (`"./plugins/php-backend-sdlc"` — already used in `/home/kravtsov/Projects/claude-plugins/.claude-plugin/marketplace.json`) or an object (`{"source":"git-subdir","url":...,"path":"plugins/...","ref":"vX.Y.Z","sha":...}`) — official marketplace pins ref+sha per release.

## 3. Gap analysis — what blocks reuse today

1. **Hardcoded project context in HIGH/MEDIUM skills** (§1.1): MongoDB repository naming (cache-management), AWS EMF/AppRunner (observability), VilnaCRM containers in workspace.dsl (structurizr), `src/User|OAuth|Shared` bounded contexts (code-organization, openapi-development). Requires the design's project-profile substitution (`.claude/php-sdlc.yml` via `/sdlc-setup`).
2. **Copy-drift already visible inside user-service**: `database-migrations` frontmatter and `implementing-ddd-architecture` line 58 claim "Doctrine ORM (MySQL for this service)" while `CLAUDE.md`/`AGENTS.md` declare MongoDB/Doctrine ODM, and `query-performance-analysis` targets MySQL/MariaDB EXPLAIN. This is direct evidence that manual skill copying between sibling repos corrupts persistence assumptions — exactly the failure the profile mechanism must eliminate.
3. **Repo-local make/script coupling**: skills reference `make ci` (52×), `make deptrac` (28×), `make ai-review-loop` (22×), `make pr-comments` (7×). Backing scripts (`ai-review-loop.sh`, `get-pr-comments.sh`, `bmad-fr-nfr-review-gate.sh`) live in `user-service/scripts/`, not in any installable artifact. The plugin must either ship them under `scripts/` (invoked via `${CLAUDE_PLUGIN_ROOT}`) or have `/sdlc-setup` verify/install make targets in the target repo.
4. **Codex-platform assumptions**: user-service `.ralphrc` defaults `PLATFORM_DRIVER=codex`; `make ai-review-loop` defaults to the Codex agent and requires Codex CLI `--output-last-message`; `AGENTS.md` headline addresses OpenAI/Codex agents. The plugin's v1 scope is claude-code driver (`bmalph run --driver claude-code`), so defaults must flip and Codex paths become optional.
5. **Governance lives in repo files, not the plugin**: the mandatory all-skills gate, thresholds, and validation-architecture rules sit in target-repo `CLAUDE.md`/`AGENTS.md`. A plugin cannot inject CLAUDE.md; `/sdlc-setup` must generate/patch these files in the target repo (or the SDLC commands must restate the gate themselves).
6. **Threshold inconsistencies to resolve during generalization**: complexity 94% (CLAUDE.md/AGENTS.md) vs 93% (ci-workflow skill); MSI "high" vs "100%" — profile should carry per-repo numbers with user-service values as defaults.

## 4. Risks

- **bmalph/claude CLI version drift**: COMMANDS.md and `_bmad/` are bmalph-generated and change across versions (bmalph 2.11.0 today); `.ralphrc` pins `CLAUDE_MIN_VERSION=2.0.76` while 2.1.170 is installed. Plugin must verify versions at `/sdlc-setup` (doctor preflight) and never vendor generated assets, otherwise `bmalph upgrade` and the plugin fight over the same files.
- **Permission modes for autonomous loops**: Ralph runs `CLAUDE_PERMISSION_MODE="bypassPermissions"`; the outer `/sdlc` loop spawning `claude` non-interactively needs equivalent grants in the target repo. Plugins cannot ship user settings — setup must document/configure `.claude/settings.json` permissions explicitly, and PERMISSION_DENIAL_MODE handling must surface rather than silently loop.
- **Token cost of the mandatory 21-skill gate**: SKILL.md bodies alone run 135–593 lines each, plus 44K REFERENCE.md-class support files, plus `REVIEW_MODE` overhead (+10–30%) and the FR/NFR gate. A naive "open every skill every feature" loop per design step 5/8 multiplies this per iteration; planning should consider applicability triage that still satisfies the gate's "Not applicable + reason" rule.
- **Target-repo variance**: ORM/MySQL vs ODM/MongoDB, REST-only vs REST+GraphQL, differing make target sets, absent Structurizr/EMF stacks. Profile detection (composer.json, doctrine config, Makefile scan) must drive skill behavior; the drift in §3.2 shows what happens otherwise.
- **CI/reviewer availability on fresh repos**: a repo born from `php-service-template` may lack the 21 workflows, CodeRabbit app installation, branch protection, and review scripts. `ci-fixer` and `pr-comment-resolver` agents need explicit degrade paths (no checks → skip-with-report; no AI reviewer → rely on local `ai-review-loop` equivalent) instead of spinning against the design's max-5-iteration guards.
- **Marketplace install-mode divergence**: local relative `source` vs pinned `git-subdir` behave differently for updates; release tagging discipline (`ref`+`sha`) is needed before external consumption.

## 6. Constraints planning must respect

- **No vendoring of bmalph/Ralph/BMAD assets** — wrap CLIs only (`bmalph init/implement/run/doctor/status/upgrade`); `_bmad/`, `.ralph/` remain bmalph-managed in target repos (design §3 external deps, §6 non-goals).
- **Root-cause-fix culture is non-negotiable**: no suppression annotations, never modify `deptrac.yaml` to pass, fix code not thresholds (AGENTS.md "Quality Improvement Guidelines", SKILL-DECISION-GUIDE line 4).
- **Thresholds never lowered**: generalized skills must keep the protected-threshold tables intact, parameterizing values per profile without weakening user-service defaults.
- **Container-only execution** in target repos (`make` / `docker compose exec php`), per AGENTS.md critical rule.
- **Every loop bounded**: 5-iteration guard per stage then escalate with a status report (design §4); honor Ralph's circuit breaker rather than auto-resetting it.
- **v1 scope**: PHP backends only, claude-code driver only, no GitHub App/hosted service (design §6).

## 7. Open questions

1. **Profile vs. rewrite mechanics**: do generalized skills read `.claude/php-sdlc.yml` at runtime (skills stay static, agent interpolates), or does `/sdlc-setup` render repo-specific skill copies into the target repo? Design implies runtime reference; affects testability and update path. Needs an architecture decision.
2. **Governance injection**: exact mechanism for getting the mandatory-gate text and thresholds into target-repo `CLAUDE.md`/`AGENTS.md` without clobbering existing content (append-managed-block? separate imported file?).
3. **Review-script portability**: ship `ai-review-loop.sh`/`get-pr-comments.sh`/`bmad-fr-nfr-review-gate.sh` in plugin `scripts/` versus requiring target repos to carry them (template-sync) — licensing/ownership is internal so either is viable; maintenance trade-off undecided.
4. **Gate cost policy**: is per-feature execution of all 18+ skills mandatory verbatim in plugin v1, or may `/sdlc-review` triage applicability first (still recording "Not applicable + reason")? Token budget differs by an order of magnitude.
5. **Permission bootstrap**: which exact permission configuration (`.claude/settings.json` allowlists vs `--dangerously-skip-permissions` invocations from Ralph drivers) `/sdlc-setup` should write for unattended target-repo runs — needs verification against current claude CLI (2.1.170) behavior.
6. **Threshold canonicalization**: which complexity value (93 vs 94) and MSI value ("high" vs 100%) become the shipped defaults — needs confirmation from user-service Makefile/CI configs during planning.
7. **bmalph compatibility floor**: minimum bmalph version the plugin supports (2.11.0 verified locally; `bmalph init` flags surface for non-interactive bootstrap on fresh repos untested here).
