# Design: `php-backend-sdlc` Claude Code Plugin

Date: 2026-06-09
Status: Approved (autonomous goal-loop; requirements derived from the /goal directive)

## 1. Problem

VilnaCRM's `user-service` carries a mature but repo-locked AI engineering setup:

- 21 Claude skills in `.claude/skills/` (DDD, API Platform, Deptrac, testing, caching, observability, docs, BMAD gates…)
- Governance docs: `AGENTS.md`, `CLAUDE.md`, `SKILL-DECISION-GUIDE.md`, `AI-AGENT-GUIDE.md`, `KERNEL-TEMPLATE.md`
- BMAD method install (`_bmad/` with bmm agents + 4-phase workflows) managed by the `bmalph` CLI
- Ralph autonomous implementation loop (`.ralph/` — drivers for claude-code/codex, circuit breaker, quality gates)
- PR governance: 20 CI workflows, CodeRabbit AI review, `make ci`, `make ai-review-loop`, BMAD FR/NFR review gate

None of this is installable elsewhere. `core-service`, `crm`, and any repo born from
`php-service-template` must copy files by hand and edit out `user-service` specifics.

**Goal:** package this setup as a repeatable, scalable Claude Code plugin that automates
PHP backend engineering end-to-end. A user installs the plugin, describes a task in a few
sentences, and the plugin drives the full SDLC: GitHub issue → BMAD planning → bmalph/Ralph
implementation with Claude CLI → multi-skill code review + BMAD FR/NFR gate → manual-style QA
verification → CI auto-fix → resolution of every AI reviewer comment → finished PR, looping
with self-criticism until everything is green and no new review findings appear.

## 2. Approaches considered

| | Approach | Trade-offs |
|---|---|---|
| A | **Thin orchestrator** — plugin ships only SDLC commands/agents, delegates to skills already present in the target repo | Smallest surface, but not repeatable: fresh repos have no skills, so the core goal fails |
| B | **Full bundle (chosen)** — one plugin shipping generalized skills + SDLC orchestrator commands + agents + BMAD/bmalph bootstrap, hosted in an org marketplace repo | Single install gives the whole capability on any PHP backend; marketplace layout leaves room to split later |
| C | **Multi-plugin marketplace** — separate plugins for skills, orchestrator, BMAD bootstrap | Maximum modularity, but 3× packaging/CI/review overhead now; nothing forces the split today (YAGNI) |

**Decision: B.** Marketplace repo `VilnaCRM-Org/claude-plugins` with the single plugin
`plugins/php-backend-sdlc/`. The marketplace structure is the scalability story: future
plugins (frontend, infrastructure) are additive.

## 3. Architecture

```
claude-plugins/                          # marketplace repo
├── .claude-plugin/marketplace.json      # lists php-backend-sdlc
├── plugins/php-backend-sdlc/
│   ├── .claude-plugin/plugin.json       # name, version, description, author
│   ├── commands/                        # SDLC slash commands (thin, delegate to skills/agents)
│   │   ├── sdlc.md                      # /sdlc "<task>" — full end-to-end loop
│   │   ├── sdlc-setup.md                # bootstrap target repo: bmalph init, project profile
│   │   ├── sdlc-issue.md                # create GitHub issue from task description
│   │   ├── sdlc-plan.md                 # BMAD planning (brief → PRD → architecture → stories)
│   │   ├── sdlc-implement.md            # bmalph implement + bmalph run (claude-code driver)
│   │   ├── sdlc-review.md               # multi-skill review + BMAD FR/NFR gate loop
│   │   ├── sdlc-qa.md                   # manual-style QA verification
│   │   └── sdlc-finish-pr.md            # CI auto-fix + resolve all AI reviewer comments
│   ├── agents/                          # subagent definitions
│   │   ├── php-implementer.md           # DDD/hexagonal implementation agent
│   │   ├── fr-nfr-reviewer.md           # BMAD spec compliance reviewer
│   │   ├── code-quality-reviewer.md     # skill-driven reviewer (deptrac/psalm/insights lens)
│   │   ├── qa-manual-tester.md          # black-box endpoint/behavior verification
│   │   ├── ci-fixer.md                  # iterates failing GitHub checks to green
│   │   └── pr-comment-resolver.md       # fetches + addresses AI reviewer comments
│   ├── skills/                          # generalized from user-service (21 skills)
│   ├── scripts/                         # setup helpers (bmalph bootstrap, profile generator)
│   ├── docs/                            # plugin docs: install, usage, SDLC loop reference
│   └── README.md
├── docs/superpowers/specs/              # this design
└── .github/workflows/                   # plugin CI (see §7)
```

### Generalization strategy (skills)

Each skill is rewritten with project specifics extracted into a **project profile**:
`.claude/php-sdlc.yml` in the target repo, generated by `/sdlc-setup` (detects composer.json,
ORM vs ODM, make targets, bounded contexts, CI thresholds). Skills reference profile keys
("the configured persistence layer") instead of hardcoded `User`/MongoDB/VilnaCRM names.
Skills that are already generic (ci-workflow, testing-workflow, quality-standards,
complexity-management, load-testing, documentation-*, openapi-development,
clean-architecture-llm, bmad-autonomous-planning) are ported with light edits.

### SDLC loop (the `/sdlc` command)

```
/sdlc "<few-sentence task>"
 1. setup-check   → /sdlc-setup if profile/bmalph missing
 2. issue         → gh issue create (acceptance criteria from task)
 3. plan          → BMAD: brief → PRD → architecture → epics/stories → readiness gate
 4. implement     → bmalph implement → bmalph run --driver claude-code (Ralph loop),
                    plus parallel subagents for independent stories
 5. review        → code-quality-reviewer + fr-nfr-reviewer + repo AI review loop;
                    refactor until a full pass yields zero new findings
 6. qa            → qa-manual-tester verifies running behavior (make start, real requests)
 7. pr            → open PR linked to issue; ci-fixer loops checks green;
                    pr-comment-resolver addresses every AI reviewer comment
 8. criticise     → re-run review+qa on final state; loop to 5 while findings exist
```

Exit condition: CI green, no unresolved AI review comments, FR/NFR gate pass, QA pass,
docs/specs/tests updated.

### External dependencies

`bmalph` (npm), `claude` CLI, `gh` CLI, Docker + make in target repos. The plugin wraps —
never vendors — bmalph/Ralph; `/sdlc-setup` verifies versions and runs `bmalph init`.

## 4. Error handling

- Every loop stage has an explicit failure path: max-iteration guard (default 5/stage) then
  escalate to the user with a status report instead of spinning.
- Ralph's circuit breaker handles implementation stalls; `/sdlc-implement` surfaces its state.
- `gh` operations check auth/permissions up front (`/sdlc-setup` preflight).
- Skills keep user-service's root-cause rule: no suppressions, never lower thresholds.

## 5. Testing

- **Plugin CI:** JSON validation (plugin.json, marketplace.json), markdown lint, shellcheck +
  bats for scripts, frontmatter schema check for every skill/command/agent.
- **QA:** install plugin from local marketplace, verify commands/agents/skills load
  (`claude plugin` + smoke prompt), dry-run `/sdlc-setup` against a `php-service-template` clone.
- **Spec gate:** BMAD FR/NFR review gate run against the plugin's own BMAD specs.

## 6. Non-goals (YAGNI)

- No rewrite of the Ralph engine or bmalph CLI.
- No non-PHP stacks in v1 (frontend/infra plugins live as future marketplace siblings).
- No GitHub App / hosted service; everything runs through the user's local CLIs.
- No support for non-Claude agents in v1 commands (skills stay markdown-portable anyway).

## 7. Delivery

Feature branch in `VilnaCRM-Org/claude-plugins`, PR linked to the bootstrap issue, CI from §5,
CodeRabbit + Claude review enabled. BMAD planning artifacts live in `specs/` of the repo.
