# FAQ

[Home](Home.md) › Operate › FAQ

Short, source-grounded answers to the questions that come up most when
adopting the php-backend-sdlc plugin. Each answer links to the page or
reference doc that covers the topic in full.

## Do I need BMAD / bmalph?

Yes — bmalph is a hard requirement, and BMAD is the planning and
implementation engine the loop is built on.

- `bmalph ≥ 2.11.0` is checked by the `/sdlc-setup` preflight; a missing
  or too-old bmalph aborts setup with a named remediation (see
  [Requirements](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/README.md)).
- A fresh repo with no `_bmad/` workspace is bootstrapped automatically
  during setup (`bmalph init`, step 2 of the
  [setup walkthrough](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/docs/setup-walkthrough.md));
  an existing `_bmad/` must be healthy (`bmalph doctor`).
- Stage 2 (`/sdlc-plan`) runs the BMAD planning chain to produce the
  six-artifact `specs/<slug>/` chain, and stage 3 (`/sdlc-implement`)
  drives implementation through bmalph/Ralph. See
  [The SDLC Loop](The-SDLC-Loop.md).

## Does it work without CI?

Yes. CI is optional and degrades cleanly. If the profile sets
`ci.provider: null` (no CI), or a PR simply has zero checks,
`/sdlc-finish-pr` skips waiting on checks and records a degrade note
instead of failing.

| Condition | Detected by | Behavior | Status |
| --- | --- | --- | --- |
| No CI workflows | `ci.provider: null` / zero checks on PR | ci-fixer skip-with-report | SUCCESS-WITH-REPORT |

This is governed by NFR-4: degrade paths never loop and never hard-fail
the run. Full table in the
[Degrade and Resilience](Degrade-and-Resilience.md) page and the
[degrade matrix](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/docs/degrade-matrix.md).

## Does it work without a reviewer app?

Yes. A reviewer app such as CodeRabbit is optional. When
`review.coderabbit: false` (no `.coderabbit.yaml` detected), the plugin's
own `ai-review-loop.sh` becomes the comment source for stage 6 — the
loop still finds and resolves review comments, it just generates them
itself instead of consuming a third-party app's.

| Condition | Detected by | Behavior | Status |
| --- | --- | --- | --- |
| No CodeRabbit / reviewer app | `review.coderabbit: false` | `ai-review-loop.sh` is the comment source for stage 6 | SUCCESS-WITH-REPORT |

See [Review and Quality Gates](Review-and-Quality-Gates.md) and
[Publishing PR Comments](Publishing-PR-Comments.md).

## Is it Symfony-only?

No. The plugin is generalized to a concrete repository through the
[project profile](Project-Profile.md) (`.claude/php-sdlc.yml`), not
hardcoded to one framework. `framework.name` is a free string (the
canonical example uses `symfony`), and framework-specific behavior is
gated on capability keys, for example:

- `framework.api_platform` — when `false`, the `api-platform-crud` skill
  is skipped.
- `framework.graphql` — toggles GraphQL handling.
- `persistence.mapper` / `persistence.engine` — select Doctrine ORM vs
  ODM and the database engine (`mysql`, `mariadb`, `postgresql`,
  `mongodb`).

In practice the plugin targets PHP backends using Doctrine and a
hexagonal/DDD layout; Symfony is the reference framework, not a hard
requirement. Every key is documented in the
[profile schema](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/docs/profile-schema.md).

## What is the profile?

The project profile is the single source of truth (FR-17) that
generalizes every command, agent, and skill to your repository. It lives
at `.claude/php-sdlc.yml`, is generated by `/sdlc-setup`
(`generate-profile.sh`), and is validated by `validate-profile.sh` before
every SDLC stage.

It captures repository facts (PHP version, framework, persistence,
bounded contexts), a `make` target map (logical target name → actual
Make target, or `null` for "capability absent"), raise-only quality
thresholds, CI/review settings, and capability flags. Quality floors may
only be tightened, never relaxed —
`validate-profile.sh` rejects any value on the wrong side of its shipped
default. Full detail on the [Project Profile](Project-Profile.md) page
and the
[profile schema](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/docs/profile-schema.md).

## Are my secrets safe?

The plugin does not exfiltrate data. The `security-auditor` agent — the
only component that actively probes — is bound by an authorized,
defensive boundary (NFR-5): it probes ONLY the profile-resolved local
service, never hosts or URLs outside it, never exfiltrates, and mutates
state only through the service's own API. Secret-scanning is part of its
static (SAST) lane: it looks for committed secrets (CWE-798) in your own
tree and reports them as findings to fix — it does not transmit them
anywhere.

Use is authorized/defensive only: the security audit is meant to be run
by the repo owner against their own service. See
[Security Audit](Security-Audit.md) and the
[security-auditor agent](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/agents/security-auditor.md).

## Does it post to PRs by default?

No. Publishing review findings to PR comments is opt-in and OFF by
default. It is gated on `capabilities.publish_pr_comments` being `true`;
when the flag is `false` or absent, the Publish step and the poster
skip-with-note (NFR-3). The default profile ships it as `false`.

The aggregate review conclusion is posted at most once per loop (via the
target mapped by `make.post_review_findings`, defaulting to the bundled
`scripts/post-review-findings.sh`), and only when the flag is enabled.
This is separate from the stage-6 comment-resolution loop, which reads
and resolves comments rather than publishing review output. See
[Publishing PR Comments](Publishing-PR-Comments.md).

## Can I run a single stage?

Yes. Each stage is also its own command, so you can run any stage
directly without the `/sdlc` orchestrator:

| Stage | Command |
| --- | --- |
| 0 setup | `/sdlc-setup` |
| 1 issue | `/sdlc-issue` |
| 2 plan | `/sdlc-plan` |
| 3 implement | `/sdlc-implement` |
| 4 review | `/sdlc-review` |
| 5 qa | `/sdlc-qa` |
| 6 finish-pr | `/sdlc-finish-pr` |

Each command carries its own Inputs/Procedure/Loop/Guard/Escalation
contract and its own `MAX_ITERATIONS=5` guard. The full-loop `/sdlc`
command simply runs these in order with gated transitions. See
[Commands](Commands.md).

## How do I resume?

Re-run the same `/sdlc` invocation. The orchestrator is resumable: on
every invocation it FIRST detects the current stage from artifacts and
resumes at the first stage (in order 0→6) whose exit condition is not yet
met — it never restarts from scratch.

Stage detection keys on durable signals, not transient stdout. Stage 1,
for example, queries open GitHub issues labeled `php-backend-sdlc` rather
than a stdout line, so a re-run cannot open a duplicate issue. A few
examples of observed state → resume point:

| Observed state | Resume at |
| --- | --- |
| profile invalid/missing or preflight stale | stage 0 → HALT: "run `/sdlc-setup`" |
| issue exists; `specs/<slug>/` incomplete or readiness not PASS | stage 2 |
| stories done; no zero-findings review-gate record | stage 4 |
| QA verdict FAIL (loop-back, stage 3 budget remaining) | stage 3 |
| finish-pr exit condition met | nothing to do → SUCCESS report |

Note that iteration counters survive loop-backs — a QA FAIL routes back
to stage 3 and consumes its remaining budget rather than resetting it.
See [The SDLC Loop](The-SDLC-Loop.md) and the
[sdlc command source](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/commands/sdlc.md).

## How do I update the plugin?

Install and updates go through the Claude Code plugin marketplace:

```bash
claude plugin marketplace add VilnaCRM-Org/claude-plugins
claude plugin install php-backend-sdlc@vilnacrm-plugins
```

While v1 ships with a relative marketplace source, installers track
`main`, so updating is pulling the latest. From the first external
consumer or v1.0.0 onward, the marketplace entry pins to a tagged
`git-subdir` source (`php-backend-sdlc-vX.Y.Z`), making installs
reproducible per release. Versioning is semver: MAJOR for a
`schema_version` bump or command-contract break, MINOR for a new
skill/command/profile key, PATCH for fixes. See the
[release process](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/docs/release-process.md).

After updating, re-run `/sdlc-setup` (or `/sdlc-setup --refresh` to
re-detect the profile) so any new governance blocks or allowlist entries
are reconciled — re-running setup is always safe.
