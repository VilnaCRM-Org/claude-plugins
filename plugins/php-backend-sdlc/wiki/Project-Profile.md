# Project Profile

[Home](Home.md) › Reference › Project Profile

The project profile is the single configuration file that adapts this
plugin to one concrete PHP backend repository. It lives at
`.claude/php-sdlc.yml` in the target repo, is generated by
[/sdlc-setup](Commands.md), and is read by every command, agent, and
skill before they run.

This page is the orientation map: what the profile is, the ten top-level
sections it carries, the raise-only thresholds rule, and how setup
creates and refreshes it. For the exhaustive key-by-key contract —
required/optional, types, defaults, and validation rules — follow the
deep link to the [full schema reference](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/docs/profile-schema.md).

## What the profile is and why

Every command, agent, and skill in the plugin is written against the
profile rather than against any one repository. Instead of hardcoding a
service name, a `make` target, a source root, or a quality floor, the
plugin reads those values from `.claude/php-sdlc.yml` at runtime. The
profile is the single source of truth that generalizes the loop to your
codebase.

Two consequences follow from this design:

- **No source literals.** Skill and command bodies must not embed
  concrete values from any specific service. Where a real value is needed
  for illustration, it appears only inside fenced examples carrying a
  `# profile-example` marker, which keeps it out of the generalization
  audit. The plugin reads `project.name`, `make.ci`, and the rest from
  your profile, so the same skill works unchanged on any PHP backend.
- **Validated before use.** The profile is checked by
  `scripts/validate-profile.sh` at the start of every SDLC stage. A
  command whose first action is validation will refuse to proceed against
  a malformed or incomplete profile, so misconfiguration surfaces as one
  clear error rather than as confusing mid-run failures.

The profile also encodes graceful degradation. A `make` target or CI
provider set to an explicit `null` means "this capability is absent
here": the plugin skips or degrades the dependent behavior with a note
instead of failing. Omitting the key entirely, by contrast, is a
validation error — the distinction between "absent capability" and
"forgot to declare" is intentional. See
[Degrade and Resilience](Degrade-and-Resilience.md) for the full matrix.

## Top-level sections

The profile opens with a single scalar, `schema_version` (must be `1`),
then groups everything else under ten top-level sections. The table below
is the orientation summary; each section's individual keys are documented
in the [schema reference](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/docs/profile-schema.md).

| Section | What it carries |
| --- | --- |
| `project` | Service `name` and GitHub `repo` (`owner/name`); used in reports and `gh` calls. |
| `php` | The PHP `version`, detected from `composer.json`. |
| `framework` | Framework `name`/`version`, plus `api_platform` and `graphql` flags that gate API skills. |
| `persistence` | The `mapper` (`doctrine-orm` \| `doctrine-odm`) and `engine` (`mysql` \| `mariadb` \| `postgresql` \| `mongodb`). |
| `architecture` | `source_root`, the list of `bounded_contexts`, and the optional `shared_context`. |
| `make` | The logical-to-actual Make target map (full local CI, boot, tests, static analysis, and more). |
| `quality` | The raise-only quality thresholds: PHPInsights scores, violation ceilings, mutation MSI. |
| `ci` | The CI `provider` plus detected `workflows` and `required_checks`. |
| `review` | Reviewer wiring: `coderabbit`, `ai_review_agents`, and `request_changes_blocking`. |
| `capabilities` | Feature toggles that gate optional skills (Structurizr, load tests, dynamic security, PR-comment publishing); `observability_emf` selects a metrics backend rather than gating a skill. |

A few sections deserve a closer look:

- **`make` is required and must be complete.** Every documented target
  key must be declared. The value is the real Make target name in your
  repo; an explicit `null` marks the capability as absent. Four
  script-substituting targets — `make.ai_review_loop`, `make.pr_comments`,
  `make.fr_nfr_gate`, and `make.post_review_findings` — fall back to their
  own bundled `scripts/` implementation rather than skipping, so the loop
  keeps working even when the repo has no such target. `make.security`
  degrades differently: when `null` the plugin runs a bundled static lane
  (Psalm taint analysis, Semgrep, `composer audit`, secret-scan) directly
  through the container, not a single substituted script.
- **`capabilities` gate optional skills.** Each boolean toggles a
  feature: `structurizr` gates architecture-sync, `load_testing` pairs
  with `make.load_tests`, `dynamic_security_testing` pairs with
  `make.start` to gate live-service probing in the security audit, and
  `publish_pr_comments` is an opt-in (default OFF) for the PR-comment
  publisher. When a gate is off, the dependent skill skips with a note —
  it never errors. See [Review and Quality
  Gates](Review-and-Quality-Gates.md) and
  [Security Audit](Security-Audit.md).
- **`ci.provider: null` means no CI.** [/sdlc-finish-pr](Commands.md)
  then skips check-waiting with a degrade note; the key must still be
  declared.

## Raise-only thresholds rule

The `quality` section ships with defaults that define the **minimum
bar**. A profile may tighten the bar but never relax it, and
`validate-profile.sh` rejects any value placed on the wrong side of its
shipped default. Two directions exist:

- **Floors (raise-only).** Score thresholds may be raised above the
  default, never lowered: `quality.phpinsights.quality`,
  `quality.phpinsights.architecture`, `quality.phpinsights.style`,
  `quality.phpinsights.complexity`, and `quality.infection_msi`.
- **Ceilings (fixed at zero).** Violation-count ceilings ship at `0` and
  may not be raised, because allowing more violations would lower the
  bar: `quality.deptrac_violations` and `quality.psalm_errors`.

The shipped defaults:

| Key | Direction | Shipped default |
| --- | --- | --- |
| `quality.phpinsights.quality` | floor (raise-only) | `100` |
| `quality.phpinsights.architecture` | floor (raise-only) | `100` |
| `quality.phpinsights.style` | floor (raise-only) | `100` |
| `quality.phpinsights.complexity` | floor (raise-only) | `94` |
| `quality.deptrac_violations` | ceiling (fixed) | `0` |
| `quality.psalm_errors` | ceiling (fixed) | `0` |
| `quality.infection_msi` | floor (raise-only) | `100` |

The practical effect: you can demand stricter complexity or a higher
mutation score for your service, and the plugin will hold you to it, but
no profile can quietly weaken the standards the loop enforces.

## How /sdlc-setup generates it

The profile is created and maintained by
[/sdlc-setup](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/commands/sdlc-setup.md),
which is the one command that does not require a profile on entry — it is
the command that creates it. The relevant steps:

1. **Preflight.** `scripts/setup-preflight.sh --report` checks the
   toolchain; on any FAIL the command aborts before touching the profile.
2. **Generate.** `scripts/generate-profile.sh` detects repository signals
   (`composer.json`, framework, persistence, source layout, CI files)
   and writes `.claude/php-sdlc.yml`. By default an existing profile is
   **never overwritten** — generation prints a detection drift diff and
   reports it instead of changing anything.
3. **Validate.** `scripts/validate-profile.sh` runs and prints one
   `VIOLATION:` line per finding. If a violation is a detection gap you
   can fix in the repository, setup enters a bounded regenerate→validate
   retry loop (`MAX_ITERATIONS=5`).

The exit condition for setup is measurable: preflight exits 0, the
profile validates clean, the governance block exists in
`CLAUDE.md`/`AGENTS.md`, and the permissions allowlist is present. See
[Getting Started](Getting-Started.md) and the
[setup walkthrough](https://github.com/VilnaCRM-Org/claude-plugins/blob/main/plugins/php-backend-sdlc/docs/setup-walkthrough.md).

### The `--refresh` flag

By default `/sdlc-setup` keeps an existing profile and shows only the
drift diff — it does not silently overwrite your file. Pass `--refresh`
to regenerate `.claude/php-sdlc.yml` from fresh detection even when one
already exists:

```bash
/sdlc-setup --refresh
```

There is one internal nuance worth knowing. Inside the validation
retry loop, regeneration always runs with `--refresh` so a corrected
repository signal actually overwrites the invalid profile — without it
each retry would be a no-op and the loop could never converge. This
in-loop refresh is not a silent overwrite: you have already seen the
`VIOLATION:` lines and are explicitly correcting the detection gap, so
the overwrite is the acknowledged remedy.

If validation still fails after the loop is exhausted, or a violation is
not fixable by re-detection, setup tells you to either edit the profile
by hand or re-run `/sdlc-setup --refresh` after correcting the
repository signals the detector reads.
